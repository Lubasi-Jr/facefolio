# Database migrations with Alembic — a reusable reference

A personal cheat sheet for how migrations work and the workflow to follow in any
project. Save this; the commands transfer to Django, Rails, Prisma, etc. — only the
command names change.

---

## The one-sentence idea

Migrations are version control for your database schema: an ordered, append-only
chain of change files, checked into git, that can rebuild or update any copy of the
database to the same structure.

---

## How it works under the hood

- Each migration is a file with an `upgrade` (apply the change) and a `downgrade`
  (reverse it). Each knows the migration before it, forming an ordered chain:
  `001 -> 002 -> 003 -> ...`
- Alembic keeps a tiny table in the database, `alembic_version`, holding the ID of
  the latest applied migration. That is how each database tracks where it stands.
- `alembic upgrade head` compares where the database is (from `alembic_version`)
  to the newest file (`head`) and runs only the migrations that are missing. Safe to
  run repeatedly: if already at head, it does nothing.

Mental model = git for schema:
- models = desired state (like working files)
- `revision --autogenerate` = commit (captures the diff as a permanent record)
- migration files = commit history
- `upgrade head` = apply the latest, replaying whatever a database is missing
- `alembic_version` = the pointer to which "commit" a database is on

---

## One-time setup (per project)

1. **Init Alembic** — scaffold the `migrations/` folder and config. Run once, ever.
   ```bash
   alembic init migrations
   ```
2. **Configure Alembic** — edit `env.py` (or `alembic.ini`) to:
   - use your project's database URL (ideally read from your app config / env)
   - import all your models so their metadata is discoverable for autogenerate

---

## The repeating loop (every schema change)

3. **Change your models** — this defines the new desired state.
4. **Generate the migration** — Alembic diffs models vs. the actual database and
   writes a new file:
   ```bash
   alembic revision --autogenerate -m "short description of the change"
   ```
5. **READ the generated migration.** Do not skip this. Autogenerate is good but not
   perfect — it can miss column renames (sees drop+add), some index/constraint
   nuances, and it will NOT add things like `CREATE EXTENSION` unless you tell it.
   Open the file, confirm it does what you intended, edit if needed.
6. **Apply it** — run the change against the database:
   ```bash
   alembic upgrade head
   ```
7. **Commit the migration file to git** so it becomes part of project history.

Compressed: **init + configure once; then forever "change models -> generate ->
read -> apply -> commit."**

## Common commands

```bash
alembic upgrade head        # apply all missing migrations (the everyday command)
alembic downgrade -1        # reverse the most recent migration
alembic downgrade base      # unwind everything back to empty
alembic current             # show which migration this database is currently on
alembic history             # show the full chain of migrations
alembic revision -m "msg"   # create an EMPTY migration to write by hand (no autogen)
```

- `head` = newest migration in the chain. `base` = empty, nothing applied.
- The database must be RUNNING and reachable when you generate (it reads current
  state) and when you apply (it writes). If it is down, both fail with a
  connection error.

---

## Golden rules (the disciplines that avoid pain)

1. **Never edit an already-applied, already-committed migration to change schema.**
   Add a NEW migration instead. Migrations are append-only, like git commits. If you
   edit an old one that production or a teammate already ran, their `alembic_version`
   already says it ran, so your edit never applies there — silent drift.
2. **Always read autogenerated migrations before applying.** Especially the first
   one, and anything involving extensions, renames, enums, or custom types.
3. **One logical change per migration**, with a clear `-m` message. Small, reviewable
   steps beat one giant migration.
4. **Commit the migration with the model change** in the same PR/commit, so code and
   schema history stay together.
5. **First migration is special**: it goes from empty to your whole starting schema.
   If you use a DB extension (e.g. pgvector), make sure `CREATE EXTENSION ...` is at
   the TOP of the first upgrade, before any table that uses that extension's types.

---

## Snapshot vs. history

- A hand-written `schema.sql` is a SNAPSHOT: the whole structure as it looks now.
  Great for documentation and standing up a fresh DB fast. It has no notion of
  change over time.
- Migrations are the HISTORY: the ordered sequence of changes that produces that
  structure, applied incrementally without losing existing data.
- Keep both if you like, but treat MIGRATIONS as the source of truth for evolving a
  real database. The snapshot is a photo; the migrations are the film.

## How a teammate syncs their database from your pushed migrations

The whole point of committing migrations: anyone can rebuild the exact schema from
nothing. When a teammate clones the repo, they do NOT hand-write any SQL and do NOT
need your `schema.sql`. They just replay the migration chain.

Their steps:

1. **Clone and install.**
   ```bash
   git clone <repo-url>
   cd <repo>/backend
   uv sync            # install deps, including alembic, from the committed lockfile
   ```

2. **Provide their own database.** They need a Postgres instance and its connection
   URL in their own `.env` (copied from `.env.example`). It can be:
   - their own local Docker Postgres (`docker compose up`), or
   - their own Supabase project.
   The database can be completely empty — that is fine, and expected.

3. **Apply all migrations.**
   ```bash
   uv run alembic upgrade head
   ```
   Because their database is empty, its `alembic_version` is unset, so Alembic runs
   the ENTIRE chain from `001` to `head` in order, building every table, index, and
   constraint step by step. Their schema now matches yours exactly.

4. **Verify.**
   ```bash
   uv run alembic current        # should show the same head revision as yours
   ```

That is the entire sync process. No SQL copy-paste, no drift.

### Staying in sync over time

- When YOU add a new migration and push it, your teammate just pulls and runs
  `uv run alembic upgrade head` again. Alembic sees their DB is behind and applies
  only the new migrations. Same command, every time.
- Everyone runs the same chain in the same order, so every database — local,
  teammate's, production/Supabase — ends up identical.
- Production (Supabase) is synced the same way: point Alembic at the production
  database URL and run `alembic upgrade head` as part of your deploy. (Some teams run
  migrations from CI/CD; the mechanism is identical.)

### Two gotchas for teammates

- If they get a connection error, the database is not running or the URL in `.env`
  is wrong — not a migrations problem.
- If they used your `schema.sql` to set the DB up manually AND then run
  `alembic upgrade head`, it will conflict (tables already exist). Pick one path:
  either replay migrations onto an empty DB (preferred for developers), OR use
  `schema.sql` for a quick throwaway/inspection DB. Do not mix them on the same DB.
